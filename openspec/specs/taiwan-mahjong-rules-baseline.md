# 台灣 16 張麻將規則 baseline

本文件定義目前 AI 與規則引擎採用的台灣 16 張麻將 baseline。
若本文件未特別說明，先以常見台灣民間桌規理解；若遇到不同桌規，必須透過規則設定覆蓋，不得直接寫死於程式。

## 基本桌規

- 玩家數：4 人
- 麻將總數：144 張
  - 萬子 36 張
  - 筒子 36 張
  - 索子 36 張
  - 風牌 16 張
  - 三元牌 12 張
  - 花牌 8 張
- 閒家起手 16 張
- 莊家起手 17 張
- 莊家先出牌
- 每回合摸 1 打 1
- 胡牌後手牌共 17 張
- 牌摸完仍無人胡牌時流局

## 合法動作規則

- 摸
  - 輪到自己回合時，從牌牆摸 1 張牌。
- 打
  - 摸牌後或整理既有手牌後，打出 1 張牌。
- 吃
  - 只能吃上家打出的牌。
  - 吃牌後形成順子，例如 `123`、`234`、`789`。
- 碰
  - 任一家打出的牌皆可碰。
  - 碰牌者手中必須已有兩張相同牌。
- 明槓
  - 以他家打出的牌完成槓子。
  - 槓後須補牌。
- 暗槓
  - 以自己手中的四張相同牌組成槓子。
  - 槓後須補牌。
- 加槓
  - 已有碰子時，再以第 4 張相同牌加成槓子。
  - 槓後須補牌。
- 胡
  - 取得合法胡牌結構時可宣告胡牌。
- 過
  - 在可宣告吃、碰、槓、胡時選擇放棄該次宣告。

### 花牌補牌

- 花牌共 8 張：春、夏、秋、冬、梅、蘭、竹、菊。
- 摸到花牌後必須立即亮出，並從牌尾補牌。
- 若補到花牌，必須持續補牌直到補到非花牌。

### 宣告優先序

- 多人同時宣告時，baseline 先採 `胡 > 槓 > 碰 > 吃`。
- 若後續桌規要求不同優先序，必須由規則設定覆蓋，不得散落在 UI 或 store。

## 胡牌成立條件

- 台灣 16 張標準胡牌結構為：
  - 五組面子
  - 一組將眼
- 可用結構示意如下：

```text
AAA
ABC
ABC
AAA
ABC
AA
```

- 順子、刻子、槓子、將皆可構成合法胡牌牌型。
- 聽牌表示只差一張即可胡牌，常見型態可包含單吊、邊張、嵌張、雙碰、多面聽。

## 台型清單

本 baseline 的 scoring 主線改採「先固定權威牌型目錄，再由 rule config 選擇 profile 與啟用規則」。

### Scoring profile

本專案第一版權威 profile 固定為兩套，可由 `Rule Config` 切換：

| profile id | 中文名稱 | 用途 |
| --- | --- | --- |
| `classic-taiwan` | 一般台數表 | 採一般台灣 16 張台數表邏輯 |
| `flower-wind-bonus` | 見花見字 | 採 GameTower 頁面中的見花見字特殊規則 |

### 權威規則落地原則

- 每一個台型在進入程式邏輯前，必須先在本文件或後續權威 spec 中補上：
  - 穩定識別 `pattern id`
  - 台型名稱
  - 適用 profile
  - 是否啟用
  - 台數值
  - 成立條件
  - 可否與其他台型疊加
  - 若屬覆蓋型規則，需列出會覆蓋哪些基礎台型
- `scoringItems` 必須使用穩定識別與對應名稱、台數、命中原因；不得只回傳一段顯示字串。
- 未完成上述資料前，不得直接把台型數值寫進 scoring code。
- 若某台型僅列為工作項目而尚未定值，程式只能保留開關或未定案狀態，不得自行猜數值。

### 第一版 catalog 納入邊界

以下項目暫不列入第一版完整練習器 catalog，原因是其本質偏房規、平台加成或連莊支付延伸，而非手牌台型判讀核心：

## 結算規則

- 自摸
  - 摸進最後所需牌張並完成胡牌時，視為自摸。
  - baseline 先記錄常見桌規為自摸三家付。
  - 實際支付金額與是否另有包牌機制，必須由桌規設定決定。
- 放槍
  - 玩家打出他人可胡之牌張時，視為放槍。
  - baseline 先記錄常見桌規為放槍者單獨支付。
  - 實際支付金額與是否一炮多響，必須由桌規設定決定。
- 莊家
  - 莊家胡牌時可連莊。
  - 閒家胡牌時，下家坐莊。
- 流局
  - 牌摸完無人胡牌時流局。
  - 流局後由原莊家續莊並重新開局。
  - 查聽、流局罰則、臭莊等延伸規則若未另行定義，不得直接寫死於程式。

## 待確認／爭議規則

以下項目目前不得直接寫死；若未有明確設定或後續 spec，實作時必須視為待確認或待定值：

| 項目 | 狀態 | 備註 |
| --- | --- | --- |
| 台型完整清單 | 進行中 | 已先固定第一批與第二批工作清單，細項仍需逐批定值 |
| 特殊胡型是否採用 | 已部分定值 | 第一版已納入天胡、地胡、八仙過海、搶槓胡；其餘特殊胡待後續批次 |
| 搶槓規則 | 已部分定值 | 先定義 `robbing-kong` 計台；完整時機與流程仍需 round flow 規格補齊 |
| 是否封頂 | 待定值 | `Rule Config` 必須保留 `maxTai` 與衝突規則設定入口，但本 baseline 尚不指定數值 |
| 過水限制 | 待確認 | 過手可否再胡或再宣告 |
| 流局是否查聽 | 待確認 | 流局後是否需要依聽牌狀態處理 |
| 花牌是否直接計分 | 已定值 | 依 profile 切換：`classic-taiwan` 採對位花，`flower-wind-bonus` 採見花見台 |
| 最低胡牌台數 | 已定值 | 採 `0 台起胡` |
| 低關聯平台加成台型 | 排除於 v1 | `幸運柴神` 不列入第一版練習器 catalog |

## 測試案例對照

- `FLOWER-REPLACE-*`：驗證摸花立即亮出、尾端補牌、補到花繼續補
- `CLAIM-PRIORITY-*`：驗證多人同時宣告時依 `胡 > 槓 > 碰 > 吃` 處理
- `WIN-STANDARD-*`：驗證五組面子加一組將眼的標準胡牌結構
- `WIN-MELD-*`：驗證副露後仍可成立標準胡牌
- `DRAW-*`：驗證牌摸完無人胡牌時流局

- `SCORE-PROFILE-*`：驗證同一手牌在不同 scoring profile 下會得到不同台型組合
- `SCORE-CONFLICT-*`：驗證互斥、覆蓋、不重複計台與封頂規則
- `PATTERN-HEAVEN-WIN-*`：驗證天胡特殊胡型的成立條件與結果摘要
- `PATTERN-EARTH-WIN-*`：驗證地胡特殊胡型的成立條件與結果摘要
- `PATTERN-BIG-THREE-DRAGONS-*`：驗證大三元特殊胡型的成立條件與結果摘要
- `PATTERN-LITTLE-THREE-DRAGONS-*`：驗證小三元特殊胡型的成立條件與結果摘要
