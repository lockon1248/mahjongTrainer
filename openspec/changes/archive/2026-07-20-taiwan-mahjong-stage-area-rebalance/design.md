## Context

使用者已多次明確指出同一件事：問題不在 header，也不在上方資訊列，而在遊戲區本身仍然太小、太擠、太空。前幾輪 child changes 先後修掉了 viewport 捲軸、過度縮小、寬度不足、高度過高與過度攤平，但目前畫面仍停在「技術上過線、視覺上不對」的狀態。

因此這次不再用單一尺度指標判斷成功，而是直接把失真拆成三個要修的畫面語意：

1. 固定舞台必須更有效利用桌機寬度，不能保留大量無意義左右留白。
2. 中央牌池與周邊玩家區要像桌面結構，不是狹長中欄加上下大資訊板。
3. 下方玩家區要保有操作可讀性，但不能再因高度或欄位撐法顯得臃腫。

## Goals / Non-Goals

**Goals:**

- 擴大實際遊戲區寬度佔比，減少桌機左右留白。
- 提高中央牌桌結構的存在感，讓中區不再顯得過窄。
- 壓縮下方玩家區的空洞感，讓它回到操作面板而不是展示板。
- 保持 header 與上方資訊列內容／樣式完全不變。
- 維持固定舞台與無主頁捲軸。

**Non-Goals:**

- 不更動 header。
- 不改上方資訊列的內容與 visual style。
- 不重寫 fixed-stage 測量演算法。
- 不在這次順手重做手機版。

## Decisions

### 以「舞台寬度利用率」和「桌面密度」作為新成功標準

之前只驗 `scale`、frame 比例與捲軸，會讓畫面落在一個仍不對勁的中間值卻被當作成功。這次新增兩個判準：

- 舞台寬度利用率：固定舞台在桌機下應吃掉更多 viewport 寬度，但不能回到滿版攤平。
- 桌面密度：中央牌池與玩家區的佔比要更接近桌面結構，不再由底部玩家區獨占過多垂直視覺。

### 需要把寬螢幕上限再往上推一段，而不是只靠 104rem

最新量測顯示在約 `2048px` 的桌機 viewport 下，固定舞台目前只佔 `81.25%` 寬度，左右各留 `192px` 空白。這證明前一次修正雖然把 1911px 桌面拉回可接受區間，但在更寬的螢幕仍過早撞到 `104rem` 上限。這次要做的是把桌機上限再往上推一段，同時加上寬螢幕驗證，避免只對單一 viewport 有效。

### 優先調整舞台寬度上限與三列比例，再調面板內密度

先修外框，再修內文。也就是：

1. `GameView` 先放寬 stage frame / scaler 的有效寬度。
2. `GameTableView` 再把三列 grid 比例與 bottom panel 內容密度拉回更平衡的桌面版型。

這樣可避免再次用面板內壓縮去硬換縮放倍率。

### 底部玩家區改成「操作優先」密度

底部玩家區是玩家最常操作區塊，應該比其他座位略大，但不能因為統計區與空白配置讓它像一塊過高的看板。這次會收斂底部面板的欄位高度、垂直間距與中央空白。

## Implementation Contract

### Behavior

- 桌機 viewport 下，固定舞台 MUST 比目前版本更有效利用寬度，明顯減少左右留白。
- 在約 `2048px` 的桌機 viewport 下，固定舞台 MUST 明顯大於目前 `0.8125` 的寬度利用率。
- 中央牌池 MUST 比目前版本更有存在感，不得繼續顯得被擠在過窄的中欄。
- 下方玩家區 MUST 保持操作可讀性，但不能繼續以過高空白感佔據主要視覺。
- header 與上方資訊列 MUST 不變。
- 仍 MUST 無主頁捲軸。

### Interface / data shape

- 不新增任何 domain state。
- 只允許修改 stage frame / scaler 寬度、桌面 grid 比例、玩家面板內容密度與必要測試。

### Failure modes

- 左右留白仍明顯偏大，失敗。
- 中央牌池仍像狹窄中欄，失敗。
- 底部玩家區仍像空洞資訊板，失敗。
- 動到 header 或上方資訊列樣式，失敗。
- 為了放大重新出現主頁捲軸或再次過度攤平，失敗。

### Acceptance criteria

- UI 測試能辨識新的 stage / table / bottom-panel 語意 class。
- Browser 驗證需同時確認：桌機下舞台寬度利用率提升、在較寬 viewport 下不再保留當前等級的邊界留白、仍無主頁捲軸、且沒有回到滿版攤平。
- 完成後需以實際瀏覽器截圖再檢查一次，不得只靠數值測試宣稱完成。
