## Context

目前對局畫面採取內容自然撐高的做法：上方摘要、中央牌桌、四家玩家區與玩家操作列都直接參與頁面流式排版。當玩家副露增加、結果摘要顯示、或桌面資訊密度提高時，整個 `game view` 會超出可視高度並產生主頁捲軸。這和麻將牌桌作為單一場景的使用模型衝突，因為玩家需要同時看見桌面摘要、四家資訊、中央牌池與自己的操作列。

這次變更只處理桌機與平板的可玩畫面收斂，不把手機直向完整可玩版納入同一輪。專案目前也沒有原生 App 能力，因此不以強制裝置旋轉作為方案前提。

## Goals / Non-Goals

**Goals:**

- 讓整個對局畫面在桌機與平板上維持單一視窗內可見，不再出現主頁捲軸。
- 讓上方摘要、中央牌桌、四家牌區與操作列作為同一個舞台一起等比縮放。
- 將超出空間的壓力優先收斂到局部區塊，而不是再把整頁高度撐開。
- 補上對應的 UI 驗證，保護固定舞台與無主捲軸語意。

**Non-Goals:**

- 不在本 change 內承諾手機直向完整可玩版。
- 不引入外部 UI 套件或新的 layout engine。
- 不修改麻將規則、算台、AI 決策或對局流程。

## Decisions

### 固定舞台包住整個對局畫面

整個 `GameView` 會改成一個固定視窗舞台，而不是讓 `GameTableView` 單獨自成高度。外層容器使用 `100dvh` 與 `overflow: hidden` 吸收視窗高度，內層保留一個固定設計尺寸的舞台，讓摘要、牌桌、玩家區與操作列一起縮放。這樣可以保證「不出現主頁捲軸」是整個對局畫面的語意，而不是只限某一塊牌桌。

替代方案是只把中央牌桌設成 `100vh` 或只限制某些區塊高度，但那只能局部止血，無法保證上方摘要與玩家區仍同時留在視窗內。

### 用等比縮放維持單一場景閱讀順序

舞台會依視窗寬高計算單一縮放比例，採用 `min(viewportWidth / designWidth, viewportHeight / designHeight)` 之類的策略，讓整個畫面等比縮放。這比對各區塊獨立使用 breakpoint 壓縮更穩，因為麻將畫面依賴固定相對位置；若每區各自重排，容易再次破壞閱讀順序。

代價是視窗較小時整體會一起變小，但這仍比讓頁面出現主捲軸更接近牌桌場景。

### 局部區塊必須先收斂自身高度責任

固定舞台不代表內容可以無上限成長。摘要區、副露區、結果區與玩家操作列仍需各自收斂高度，必要時由內部布局吸收密度，避免舞台內容本身溢出固定設計尺寸。這次 apply 階段應優先清理目前最容易把版面撐高的區塊，例如玩家副露列表與結果摘要容器。

替代方案是單靠整體縮放吞掉所有高度成長，但這會讓極端情況下字與牌縮得過小，且不能保證長期穩定。

### 手機先維持非主目標裝置處理

這次 change 的驗證與 acceptance 以桌機／平板為主，不承諾手機直向完整可玩，也不實作強制橫向。apply 階段可以保留現有手機退化顯示，或只做最小保護避免完全壞版，但不得把範圍擴大成一整套手機專用畫面重做。

## Implementation Contract

### Behavior

- 使用者開啟對局畫面時，桌機與平板情境下整個對局內容 MUST 在單一視窗內可見，不得因摘要、副露、玩家區或結果摘要高度增加而出現主頁垂直捲軸。
- 對局摘要、中央牌桌、四家玩家區與玩家操作列 MUST 作為同一個視覺舞台同步縮放，而不是分別溢出或各自脫離舞台。
- 增加副露、顯示下一局按鈕、或顯示結果摘要時，畫面 MUST 維持同一舞台內的可見性，超出空間的處理由局部區塊承擔。

### Interface / data shape

- `GameView` 需要擁有一個能代表固定舞台尺寸與縮放狀態的視圖層結構；若新增 UI-only state，必須明確屬於 view/layout state，而不是 domain state。
- `GameTableView` 的既有 props 契約不應因這次 change 被迫改成規則驅動之外的新 domain 欄位；若需要額外 layout 資訊，應保持在 view 層內部或由 UI-only props 傳遞。
- 樣式層需要有清楚的舞台容器 class，讓 reviewer 能定位哪個元素負責 `100dvh`、哪個元素負責縮放、哪個元素負責局部溢出收斂。

### Failure modes

- 若視窗太小導致舞台縮放後可讀性下降，這次 change 接受縮小但不接受主頁出現捲軸。
- 若某一局部區塊仍可能產生內部溢出，它必須被限制在舞台內部，不得再把整頁高度撐開。
- 手機直向若仍無法提供舒適可玩版，不視為這次 change failure，但不得讓桌機／平板回歸到主頁可捲動狀態。

### Acceptance criteria

- UI / component 測試能證明固定舞台容器存在，且主要操作區仍可渲染。
- 至少一條真實流程驗證必須涵蓋副露或結果摘要存在時，畫面仍維持單一視窗語意。
- apply 階段完成後，browser E2E 或等效真實流程驗證必須確認桌機尺寸下頁面沒有主捲軸回歸。

### Scope boundaries

- In scope：`GameView`、`GameTableView`、相關樣式與對應 UI 驗證。
- Out of scope：手機強制橫向、原生 App 行為、AI／規則／算台修改、整體資訊架構大改版。

## Risks / Trade-offs

- [舞台縮放讓文字與牌變小] → 先以桌機／平板為主要驗證目標，並搭配局部區塊高度收斂，避免縮放成唯一吸收手段。
- [固定舞台導致某些區塊內容被裁切] → 對最容易成長的區塊設定明確內部布局責任，必要時用區塊內處理而非整頁捲動。
- [實作只修 CSS 表象，沒有保住真實流程] → 以 UI 測試與 browser E2E 驗證無主捲軸與主要操作仍可達成，避免只看靜態快照。
