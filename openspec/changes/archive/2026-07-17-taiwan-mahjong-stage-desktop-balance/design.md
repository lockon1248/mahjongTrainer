## Context

最新截圖已經證明上一輪又走到另一個極端：在更大的桌機螢幕下，固定舞台不再縮放，`game-stage-scaler` 直接回到 `scale = 1`，搭配偏大的舞台上限與過度水平展開的玩家資訊格，導致整個桌面被攤平成近乎滿版的比例，出現大量不自然的橫向空白。

這次需求很明確：不再問方向，直接把桌機比例拉回正常。上方資訊列仍然不動；要修的是牌桌在大桌機 viewport 下的比例平衡。

## Goals / Non-Goals

**Goals:**

- 避免桌機大視窗下的牌桌過度滿寬攤平。
- 保留上一輪已解掉的「不要縮在中央太小」成果。
- 維持 header 與上方資訊列不變。
- 維持固定舞台與無主頁垂直捲軸。

**Non-Goals:**

- 不回退到之前過小的舞台。
- 不修改 header 或上方資訊列內容／樣式。
- 不重寫整個 fixed-stage 演算法。
- 不在本 change 內重做手機版。

## Decisions

### 桌機上限比例要收斂，而不是讓 `scale = 1` 就直接滿寬

上一輪的問題在於，一旦內容高度被壓到足夠低，桌面在大桌機下就不再縮放，導致整個舞台直接以過大的最大寬度滿寬展開。這次要收斂大桌機下的舞台最大寬度，保留放大感，但禁止無限制攤平。

### 玩家資訊格要回到較均衡的桌面閱讀比例

四欄水平資訊格雖然壓低了高度，但在大桌面下會把每個玩家面板撐成很空的橫向長條。這次要把玩家統計區拉回較均衡的資訊密度，讓面板看起來像桌面區塊，而不是被拉寬的資料列。

### 這次的成功標準是「正常桌面比例」

不是越大越好，也不是只要 scale 高就好。這次要同時滿足：不縮小、不攤平、仍留在固定舞台內。

## Implementation Contract

### Behavior

- 在大桌機 viewport 下，牌桌 MUST 不再過度滿寬攤平。
- 玩家面板與中央牌池 MUST 維持合理桌面比例，避免出現過度空洞的水平展開。
- 調整後 MUST 仍明顯大於最初過度縮小的版本。
- 調整後 MUST 繼續維持無主頁垂直捲軸。

### Interface / data shape

- 不新增 domain state。
- 只允許修改舞台寬度上限、桌面 grid 比例與玩家資訊密度。
- 驗證必須同時覆蓋固定舞台可視尺寸與桌機比例平衡，而不是只驗其中一邊。

### Failure modes

- 若畫面又回到過小縮在中央，失敗。
- 若畫面仍像最新截圖一樣過度攤平，失敗。
- 若為了修比例而動到上方資訊列，失敗。
- 若重新出現主頁捲軸，失敗。

### Acceptance criteria

- UI 測試能辨認玩家面板已回到較均衡的資訊格語意。
- Browser 驗證需確認大桌機下舞台不會滿寬攤平，且仍維持固定舞台可視放大效果。
- 完成後需直接檢查實際截圖，確認不再出現當前這張圖的失真比例。

### Scope boundaries

- In scope：`src/views/game/GameView.vue`、`src/views/game/components/GameTableView.vue`、對應 Vitest、對應 Playwright smoke、主線 board 同步。
- Out of scope：header / summary 重排、手機版重做、fixed-stage engine 全重寫。

## Risks / Trade-offs

- [桌面比例修回來後又變太小] → 用截圖與 browser 指標一起驗證，不只看其中一邊。
- [資訊格回收後高度再次推高] → 以桌機比例平衡為主，必要時採用較保守的上限寬度，避免再靠極端密度換取尺寸。
- [scope drift] → 限制只改舞台上限、grid 比例與資訊密度。
