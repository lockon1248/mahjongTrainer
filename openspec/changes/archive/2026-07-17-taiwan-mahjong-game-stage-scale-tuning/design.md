## Context

`taiwan-mahjong-fixed-viewport-stage` 已讓對局畫面在桌機／平板下維持單一固定視窗舞台，並用縮放避免主頁捲軸回歸。但使用者實際檢視後回報另一個明顯問題：雖然不捲動了，遊戲主舞台仍顯得過小，四周留白太多，整個畫面像被縮在中間。

這次調整的前提已經明確：保留全站 header，也暫時不重做上方資訊列的內容與樣式，只先把遊戲區放大，讓牌桌與玩家區更接近可用視窗空間。

## Goals / Non-Goals

**Goals:**

- 在不恢復主頁捲軸的前提下，放大遊戲主舞台的實際視覺佔比。
- 保留全站 header 與現有資訊列內容／樣式結構，不在本 change 內做資訊列重設計。
- 收斂外層 shell、page 容器與 stage frame 的重複留白，讓固定舞台更貼近視窗邊界。
- 用 UI 測試與 browser 驗證保護「放大遊戲區但不回到小模型感」這個方向。

**Non-Goals:**

- 不修改 header 內容或導覽資訊架構。
- 不修改上方資訊列的欄位內容、卡片樣式或資訊分組。
- 不重做牌桌內部座位資訊架構、配色主題或手機專用版。

## Decisions

### 保留固定舞台，優先調整外層可用空間分配

這次不推翻固定舞台機制，而是調整它在整頁中的尺寸策略。問題不在「有沒有縮放」，而在外層 `app shell`、`game view`、`stage shell` 共同吃掉太多可用空間，導致最終舞台即使不捲動也太保守。

因此 apply 階段優先調整外層 padding、header 以下的剩餘高度分配與 stage 最大寬高，而不是先重寫桌面內部排版。

### 不碰資訊列內容／樣式，只讓它跟著更大的舞台一起長大

使用者已明確要求「資訊列的樣式內容先不動」。所以這次只能改舞台尺度，不能順手把資訊列重構成一條狀態列或新的資訊架構。這可避免 scope drift，也讓之後若要重做資訊列，可以獨立開下一份 child change。

### 外框留白保留，但要從『大面積空白』改成『有邊界感的呼吸』

這次不是做滿版，而是保留一圈外框空間，但把遊戲舞台放大約一圈，讓留白變成邊界，而不是讓中央內容像小模型。也就是說，版面目標是「留白仍存在，但不再是主要視覺面積」。

## Implementation Contract

### Behavior

- 遊戲頁在桌機／平板下 MUST 保留全站 header 與現有資訊列內容／樣式，但遊戲主舞台 MUST 比前一版明顯放大，減少四周大面積留白。
- 放大後的遊戲頁 MUST 繼續維持無主頁垂直捲軸，不得因為舞台放大而回退到可捲動頁面。
- 玩家牌區、中央牌池與結果摘要在視覺上 MUST 更接近可用視窗邊界，而不是維持縮在中間的小舞台感。

### Interface / data shape

- 這次 change 不引入新的 domain state；若需要調整尺度，僅允許修改 view/layout 層的容器尺寸、padding、最大寬高與縮放策略。
- `GameView` 與 `App.vue` 需要保有清楚的 container class，讓 reviewer 能辨認哪些元素負責 header 佔位、哪些負責遊戲舞台放大。

### Failure modes

- 若放大後重新出現主頁捲軸，視為失敗。
- 若為了放大遊戲區而偷偷改動資訊列內容／樣式，也視為超出 scope。
- 若遊戲區雖然 technically 放大，但仍保留明顯過量留白，則不符合本 change 目標。

### Acceptance criteria

- UI 測試能證明固定舞台結構仍存在，且遊戲頁容器採用新的較大舞台佈局。
- Browser 驗證需再次確認放大後沒有主頁捲軸。
- 至少一條流程驗證需覆蓋玩家區或結果摘要存在時，遊戲區仍維持放大後的固定舞台語意。

### Scope boundaries

- In scope：`App.vue`、`GameView.vue`、必要的 stage 容器尺寸策略與對應驗證。
- Out of scope：header 內容重排、資訊列內容／樣式重設計、牌桌內部資訊架構大改、手機專用版。

## Risks / Trade-offs

- [放大舞台後重新逼近捲動臨界點] → 用 browser 驗證鎖住無主頁捲軸，避免只看靜態視覺。
- [只調外層尺度但使用者仍覺得擁擠] → 本 change 僅處理「太小與留白過多」；若之後仍需壓縮資訊列，再獨立開下一份 child change。
- [scope drift 到資訊列重設計] → 設計與 tasks 明確寫死「資訊列內容／樣式不動」，apply 階段不得順手擴大範圍。
